Chaîne d'authentification RDP — Kerberos

VIp-coop · F5 niveau 4 · SPN TERMSRV/VIp-coop créé · pool de 5 backends
Kerberos·⚠ intermittent (1/5) ou nul (0/5) — non supporté

Le décor : le client (mstsc) vise le nom virtuel VIp-coop.zres.ztech. Le F5 est un pur relais TCP (L4) qui transmet les octets vers un backend d'un pool de 5 sans rien terminer. Nouveauté : on a bien créé le SPN TERMSRV/VIp-coop.zres.ztech. Le KDC émet donc maintenant un ticket — mais scellé à la clé d'un seul compte. La question devient : que se passe-t-il une fois le ticket relayé jusqu'au backend, et comment l'hôte le perçoit-il ? La réponse tient toujours dans une seule clé de chiffrement — et dans le backend sur lequel le F5 tombe.

Le cœur du problème

La clé qui chiffre le ticket

choisis un scénario

Kerberos ne chiffre pas la session RDP (ça, c'est TLS/CredSSP). Mais il authentifie au moyen de chiffrement : le KDC chiffre la partie utile du ticket (clé de session + PAC) avec la clé long-terme du compte qui porte le SPN. L'hôte qui reçoit la connexion doit posséder exactement cette clé pour l'ouvrir. La preuve d'identité, côté serveur, est ce déchiffrement. Règle simple : la couleur de la clé qui verrouille doit être identique à celle de la clé qui déverrouille.

Pourquoi c'est structurel

Trois blocages qui se cumulent

BLOCAGE 1

Le SPN est unique

TERMSRV/VIp-coop.zres.ztech ne peut exister que sur un seul compte de toute la forêt. Même créé, il ne désigne qu'une identité — jamais « le pool ».

BLOCAGE 2

Il faut posséder la clé

L'hôte qui répond réellement doit détenir la clé ayant chiffré le ticket. Chaque backend a sa propre clé machine : si le F5 route vers un autre nœud → AP_ERR_MODIFIED.

BLOCAGE 3

RDP = identité machine

TermService authentifie les connexions avec le compte d'ordinateur (LocalSystem), pas un compte arbitraire. Aucun moyen supporté de lui faire ouvrir les tickets d'un compte de service ou d'un gMSA.

Les seules issues

Ce qui ferait réellement marcher Kerberos

… au prix d'un changement de design
OPTION A

Backend unique

On pose TERMSRV/VIp-coop et HOST/VIp-coop sur le compte machine de l'unique hôte (setspn -S). Le ticket pour la VIP devient déchiffrable par ce host.

Compromis : un seul backend — plus de répartition ni de HA.
OPTION B

F5 APM + KCD (S4U2Proxy)

Le F5 termine la connexion, pré-authentifie l'utilisateur, puis obtient lui-même un ticket vers le vrai SPN du backend (TERMSRV/MachineDest) au nom de l'utilisateur, via délégation contrainte.

Compromis : fini le L4 pur et le CredSSP de bout en bout.
OPTION C

RD Connection Broker

La brique native Microsoft : le client est redirigé vers l'hôte de session réel, et Kerberos s'opère contre le vrai nom de cet hôte, pas contre la VIP.

Compromis : ce n'est plus un simple VIP L4 sur F5.
Côté capture

Dans Wireshark — ce que vous devez voir

2 captures · côté client + côté backend

Maintenant que le SPN existe, la capture côté client raconte une histoire trompeuse : l'ouverture TCP vers la VIP, la négociation RDP (X.224), l'échange Kerberos sur le port 88 avant le moindre paquet TLS — et cette fois le TGS-REP réussit (le ticket est émis !). Puis le handshake TLS, et seulement des Application Data opaques.

Le piège a changé de place : dans l'ancien scénario « sans SPN », l'échec (erreur 7) était en clair sur le port 88. Désormais le ticket part bien, l'AP-REQ est présenté à l'intérieur de TLS, et c'est là — invisible en clair — que le backend répond AP_ERR_MODIFIED. Côté client, vous voyez un Kerberos « réussi » puis un repli NTLM inexpliqué. Pour voir le « pourquoi », il faut capturer sur le backend (2ᵉ tableau).

« Kerberos se fait donc avant TLS ? »

Oui — pour la partie que vous voyez. Il faut distinguer deux sous-étapes que l'on confond souvent :

1 · Récupérer le ticket auprès du KDC (TGS-REQ / TGS-REP sur le port 88). C'est une conversation séparée avec le contrôleur de domaine, qui n'a pas besoin du canal TLS vers le serveur RDP. Windows la déclenche dès l'initialisation du contexte de sécurité (InitializeSecurityContext), donc avant d'ouvrir TLS. → c'est exactement ce que montre votre capture, et avec le SPN créé, elle aboutit.

2 · Présenter le jeton au serveur RDP (l'AP-REQ, à l'intérieur de SPNEGO/CredSSP). Celui-là passe dans le tunnel TLS, après le handshake. Cette fois il existe (trame 14) — mais le backend qui le reçoit n'a pas la bonne clé : il renvoie KRB_AP_ERR_MODIFIED dans TLS (trame 15, opaque), et le client bascule sur NTLMv2 — toujours dans TLS.

Autrement dit : l'obtention du ticket précède TLS (et réussit) ; la présentation du ticket — et son échec — se font dans TLS. Deux moments différents, et seul le second porte désormais le problème.

Pourquoi deux TGS-REQ ? Le poste est dans ZCAM.ZTECH mais la cible est dans ZRES.ZTECH : son KDC (10.20.0.10) ne peut pas émettre le ticket d'un autre domaine, il renvoie un referral vers ZRES (trame 7). Le client repose alors sa question au KDC de ZRES (10.40.0.10, trame 8), qui cette fois émet le ticket (trame 9) — le SPN existe désormais. Et c'est là que le piège se referme : côté client, Kerberos a « réussi ». L'AP-REQ part bien (trame 14) mais à l'intérieur de TLS ; le rejet AP_ERR_MODIFIED du backend (trame 15) est lui aussi encapsulé dans TLS — kerberos comme ntlmssp n'y montrent rien sans déchiffrement (clé privée serveur si RSA, ou SSLKEYLOGFILE). Conclusion : cette capture seule ne dit pas pourquoi ça bascule en NTLM — elle prouve juste que le ticket a été émis. Pour voir la cause, il faut capturer sur le backend (tableau ci-dessous) ou lire son journal de sécurité. Si le poste n'a pas de TGT en cache, un AS-REQ / AS-REP (msg_type 10 / 11) précède encore tout cela.

Le piège des « erreurs en clair ». Sur le backend on voit beaucoup de Kerberos en clair — mais presque tout est le ménage normal de C, pas l'échec RDP. L'AS-REQ → KRB-ERROR (25) (trames 11-12) est le tâtonnement de pré-authentification standard de toute machine Windows : elle demande sans pré-auth, le DC réclame un horodatage chiffré, elle recommence. Le bindRequest LDAP (trame 17) est l'ouverture de session de C vers l'annuaire. Ni l'un ni l'autre n'est la cause. Le vrai échec — l'AP-REQ relayé que C ne sait pas déchiffrer — s'est produit dans TLS (trame 8) : sur le fil il reste opaque. La seule trace réseau lisible est indirecte : le NetrLogonSamLogonEx (trames 19-20) prouve qu'on a basculé en NTLM. Pour le voir noir sur blanc, lisez le journal Sécurité de MachineDest-C : un 4625 (échec) ou un 4624 avec Package = NTLM, et le canal Microsoft-Windows-Security-Kerberos pour KRB_AP_ERR_MODIFIED.

Et si le F5 était tombé sur MachineDest-A ? La trame 8 se déchiffrerait avec la clé de A → aucun repli, pas de Netlogon NTLM, juste un 4624 avec Package = Kerberos. C'est précisément ce 1/5 qui rend la panne intermittente.

Les trois choses à comprendre dans ces captures
L'ORDRE

Kerberos avant TLS, AP-REQ dans TLS

Le ticket est demandé au KDC dès l'init du contexte → avant le Client Hello (trames 6→9, en clair). L'AP-REQ, lui, est présenté après le handshake, dans le tunnel (trame 14). Cette fois le ticket existe : Kerberos « réussit » côté client.

TRAMES 6 → 9

Un renvoi (referral) puis succès

Le double TGS-REQ est normal : la cible ZRES.ZTECH n'est pas le domaine du poste (ZCAM.ZTECH). Le premier KDC renvoie vers ZRES (7), le client réitère (8), et ZRES émet le ticket (9) — scellé à la clé d'un seul compte.

TRAMES 14-15 · BACKEND

AP_ERR_MODIFIED, caché dans TLS

Le backend qui reçoit n'a pas la bonne clé → KRB_AP_ERR_MODIFIED, mais à l'intérieur de TLS (opaque sur le fil). Côté client : Kerberos OK puis NTLM « mystérieux ». La preuve est sur le backend (journal de sécurité) ou via le Netlogon du 2ᵉ tableau.

Attention à l'endroit où vous capturez
① SUR LE POSTE CLIENT

Le ticket part, l'échec est opaque

Vous voyez l'aller-retour Kerberos (88) réussir, puis du Application Data TLS et un repli NTLM — sans la raison. Utile pour confirmer que le ticket est bien émis, insuffisant pour diagnostiquer le repli.

ENTRE F5 ET BACKEND

SNAT + persistance visibles

Les octets arrivent SNATés depuis l'IP du F5 (pas le client), vers un backend du pool. On y lit la persistance source-address — donc quel backend a reçu la session — mais toujours pas le contenu (TLS).

② SUR LE BACKEND

Le seul endroit où voir la vérité

C'est ici qu'il faut capturer. Le Netlogon (MS-NRPC) trahit le repli NTLM ; surtout, le journal Sécurité du backend porte le 4625/KRB_AP_ERR_MODIFIED puis le 4624 NTLM — la cause, noir sur blanc.

Pour bien visualiser

Le nom n'est pas une adresse — l'histoire des couleurs de clé

« Si le ticket porte bien TERMSRV/VIp-coop, pourquoi un autre serveur ne l'accepte pas ? »

Parce que ce nom n'est pas une adresse de livraison. C'est seulement une consigne donnée au KDC : « scelle ce ticket avec le cadenas du compte qui porte ce nom ». À l'arrivée, le serveur ne lit pas le nom pour décider — il essaie juste d'ouvrir le ticket avec sa propre clé. S'il a la bonne clé, ça s'ouvre ; sinon, non — quel que soit le nom écrit dessus.

L'image de la poste : l'étiquette dit « ferme ce colis avec le cadenas du serveur A ». Le colis n'est pas remis « au nom marqué dessus », il est scellé avec un cadenas précis. Le serveur B reçoit bien un colis marqué « VIP », mais il n'a pas la bonne clé — l'étiquette ne l'aide pas à l'ouvrir.

D'abord : c'est quoi « une couleur de clé » ?

Chaque machine du domaine possède une clé secrète qui lui est propre (dérivée du mot de passe de son compte d'ordinateur). Personne ne la partage. Dans ce schéma, je représente chaque clé par une couleur : même couleur = même clé (on peut ouvrir), couleur différente = clé incompatible (on ne peut pas). Le KDC, lui, connaît la clé de toutes les machines — il peut donc fermer un ticket avec la couleur de n'importe laquelle. Le cadenas prend alors la couleur de la clé qui l'a fermé, et seule une clé de la même couleur le rouvre.

Les deux façons de poser le SPN

Schéma 1 — SPN sur une machine du pool (MachineDest-A)

intermittent Kerberos : 1/5
Le KDC scelle le ticket avec la clé de MachineDest-A (cadenas vert). Seul A sait l'ouvrir. Mais le F5 répartit sans connaître Kerberos : statistiquement 1 connexion sur 5 tombe sur A et aboutit, les 4 autres renvoient AP_ERR_MODIFIED et basculent en NTLM. → panne intermittente, la plus pénible à diagnostiquer (« ça marche une fois sur cinq »).

Schéma 2 — SPN sur un compte hors pool (svc-vip / gMSA)

jamais en Kerberos Kerberos : 0/5
Le SPN est porté par un compte de service ou gMSA (svc-vip$), pas par une machine. Le ticket est scellé avec une clé (sarcelle) qu'aucun backend ne possède : les 5 hôtes n'ont que leur propre clé machine. Résultat 0/5, repli NTLM systématique. Paradoxalement plus « stable » que le schéma 1… mais Kerberos n'aboutit jamais.
vert = clé de MachineDest-A bleu = MachineDest-B ambre = MachineDest-C rouge = MachineDest-D violet = MachineDest-E sarcelle = compte de service / gMSA (hors pool)

Règle : un cadenas a la couleur de la clé qui l'a fermé, et il ne s'ouvre qu'avec une clé de la même couleur. Le F5 (niveau 4) ignore ces couleurs — il distribue selon la persistance réseau. D'où 1/5 (schéma 1, SPN sur un backend) ou 0/5 (schéma 2, SPN hors pool).

À retenir
LE NOM

Une consigne, pas une adresse

TERMSRV/VIp-coop dit seulement au KDC quelle clé utiliser pour fermer le ticket. Le serveur qui reçoit ne s'en sert pas pour décider : il tente sa propre clé.

LA CLÉ

Elle ne se partage pas

Chaque machine garde sa clé. Un ticket fermé avec la clé de A ne s'ouvre que chez A. Sur B, autre clé → AP_ERR_MODIFIED, même nom inscrit.

DONC, AVEC 5 BACKENDS

1/5 si dans le pool, 0/5 sinon

Le SPN est unique → une seule clé. SPN sur un backend → Kerberos n'aboutit que pour lui (1/5, intermittent) ; SPN hors pool → 0/5, NTLM systématique. Pour un Kerberos fiable : RD Connection Broker ou APM + KCD.
