01
Contexte technique & Topologie

Architecture cible : un poste source Windows 11 25H2 (domaine zcam) initie une connexion RDP via une VIP F5 BIG-IP en TCP bridge (objet réseau dans le domaine zres) vers un poste cible Windows 11 23H2 (domaine zcam). La connexion échoue, sans trafic visible entre la VIP et la cible dans la trace réseau.

VUE :
🎯
Particularité critique de cette architecture Le client est en 25H2 (post-durcissement avril 2026 / KB5083769) tandis que la cible est en 23H2. Cette asymétrie de versions, combinée à l'intercalation d'une VIP F5 dans un domaine tiers (zres) sans trust documenté, génère une cascade de problèmes potentiels touchant la résolution DNS, l'enregistrement SPN, la négociation Kerberos cross-realm, le fallback NTLM (potentiellement bloqué), et la validation du certificat serveur RDP.
🖥️
Source — Win11 25H2 (zcam)
Client durci. CredSSP Protected Mode actif. NLA enforced. TLS 1.2+ obligatoire. Bloque les clients NTLMv1 et les anciens CredSSP. Demande un cert RDP avec EKU "Server Authentication" + "Remote Desktop Authentication".
⚖️
Intermédiaire — VIP F5 (zres)
TCP Bridge mode = le F5 fait du forwarding L4 sans inspection TLS. Le certificat installé n'est utile que si un Client SSL profile est attaché à la VIP. Sinon : décoration sans effet sur le flux RDP.
🎯
Cible — Win11 23H2 (zcam)
Pré-durcissement. Accepte CredSSP v3, TLS 1.0+. SPN par défaut : TERMSRV/<hostname>.zcam.local. Présente son propre certificat RDP auto-signé sauf si un cert custom est déployé.
⚠️
Symptôme clé
Aucun trafic entre VIP et cible dans la trace. Cela signifie que la négociation casse côté client-F5 avant que le F5 ne forward au backend. C'est cohérent avec un échec NLA/CredSSP au niveau application, pas avec un problème réseau pur.
02
Fonctionnement du protocole RDP en 23H2 (cible)

Le protocole RDP en version 23H2 correspond à la version pré-durcissement d'avril 2026. CredSSP v3-v5 supporté, TLS 1.0+ toléré, NLA optionnel et NTLM autorisé en fallback. Cette section détaille le handshake complet tel qu'implémenté côté cible.

Stack protocolaire RDP en 23H2
COUCHES :
Caractéristiques du RDP 23H2
🔓
CredSSP plus permissif
EncryptionOracleRemediation par défaut = Mitigated (2). Accepte clients SHA-1. Compatible avec CredSSP v3 et antérieurs. Le fallback NTLM est possible.
🔐
TLS 1.0 / 1.1 tolérés
Par défaut, le serveur accepte TLS 1.0, 1.1 et 1.2. Les algorithmes de chiffrement legacy sont activés. Negotiate cipher suite incluant 3DES possible.
🎫
NTLMv2 en fallback
Si Kerberos échoue (SPN absent, KDC injoignable, cross-realm sans trust), le client tombe en NTLMv2 sans avertissement bloquant.
📜
Certificat self-signed accepté
Le serveur génère un certificat self-signed à l'installation. L'utilisateur reçoit un avertissement mais peut cocher "Don't ask again" pour bypasser.
03
Fonctionnement du protocole RDP en 25H2 (source — client durci)

Le client 25H2 applique le durcissement KB5083769 (avril 2026). Plusieurs comportements changent radicalement : CredSSP Protected Mode (SHA-256 obligatoire), NLA strictement enforced, NTLM v1 bloqué, TLS 1.0/1.1 rejetés, et nouvelle validation stricte du certificat serveur RDP.

COUCHES :
🚨
Différence clé 23H2 → 25H2 dans VOTRE scénario En 23H2, si le SPN Kerberos est absent ou le trust manquant, le client tombait silencieusement en NTLM et la connexion réussissait. En 25H2, ce fallback est verrouillé par défaut : NTLMv1 bloqué, NTLMv2 cross-realm restreint, et CredSSP Protected Mode refuse de poursuivre. Résultat : la connexion casse côté client AVANT que le F5 ne forward au backend → cohérent avec votre trace réseau vide.
04
Architecture F5 BIG-IP — Modes TCP Bridge, SSL Bridge, Passthrough

La compréhension fine du mode de fonctionnement F5 est essentielle. Le terme "TCP bridge" recouvre plusieurs réalités différentes. Cette section explique chaque mode et identifie pourquoi le certificat installé sur la VIP n'apparaît dans aucun message d'erreur.

💡
Pourquoi votre cert F5 n'apparaît dans aucun message d'erreur Dans un montage TCP bridge / SSL passthrough (votre cas le plus probable), le F5 ne fait que du forwarding L4. Le certificat que vous avez installé sur la VIP n'est jamais présenté au client RDP, car aucun Client SSL profile ne le référence. Le handshake TLS se fait end-to-end entre le client 25H2 et la cible 23H2 : c'est le certificat de la cible qui est validé. D'où l'absence de mention du cert F5 dans le message d'erreur — il est invisible pour le client.
Quel cert valide réellement le client 25H2 ?
Mode F5Cert présentéCN attendu par le clientCas d'échec
TCP Bridge / Passthrough Cert de la cible 23H2 vip.zres.lan (nom de connexion) Cert cible a CN=host.zcam ≠ vip.zres.lan → avertissement OU rejet
SSL Bridge (terminate) Cert du F5 vip.zres.lan RDP ne supporte pas la rupture TLS en milieu de chaîne → CredSSP échoue
SSL Bridge + Proxy SSL Cert du F5 visible, key de la cible utilisée vip.zres.lan ou host.zcam F5 doit avoir la clé privée du serveur cible — rarement le cas
05
Analyse approfondie — Causes racines possibles

Synthèse des trois symptômes observés et identification des causes les plus probables, classées par ordre de probabilité décroissante.

Cause #1 — Échec Kerberos cross-realm (probabilité TRÈS ÉLEVÉE)
🎯
Diagnostic principal Le client 25H2 résout vip.zres.lan et tente d'obtenir un service ticket Kerberos pour le SPN TERMSRV/vip.zres.lan. Ce SPN n'existe ni dans zcam (client) ni dans zres (VIP n'est pas un objet AD). Sans trust déclaré entre les forêts, aucun referral n'est possible. Le client échoue avec KDC_ERR_S_PRINCIPAL_UNKNOWN. Le fallback NTLM est désormais restreint sur 25H2 → CredSSP échoue avant tout MCS Connect → aucun trafic ne part vers la cible.
Cause #2 — F5 health monitor en échec (probabilité MOYENNE)

Si le monitor F5 ne valide pas correctement la cible (par exemple un monitor HTTP au lieu de TCP-3389), le pool member peut être marqué DOWN. Dans ce cas, la VIP rejette les nouvelles connexions sans forwarder → trace réseau vide entre VIP et cible.

Cause #3 — Cert F5 attaché à un Client SSL profile mal configuré (probabilité MOYENNE)

Si quelqu'un a ajouté un Client SSL profile à la VIP en pensant que c'était nécessaire pour le cert, le F5 essaye de terminer TLS. Or le client RDP attend une chaîne CredSSP continue. Le handshake TLS peut réussir, mais CredSSP échoue car la session TLS ne contient pas la bonne identité serveur. Résultat : connexion coupée côté F5.

Cause #4 — Pas de relation d'approbation entre zcam et zres (probabilité ÉLEVÉE — à vérifier)

Question explicite. Sans trust :

  • Les DC zcam ne peuvent pas demander un referral vers les DC zres pour résoudre un SPN.
  • Le client peut résoudre le DNS de vip.zres.lan (via forwarder), mais ne peut pas authentifier auprès du domaine zres.
  • Toutefois, comme la VIP n'est PAS le serveur final, ce point a moins d'impact qu'il n'en a l'air — si on règle le SPN dans zcam (sur le compte machine de la cible), le trust devient inutile.
Cause #5 — Cert sur cible avec mauvais Subject Alternative Name (SAN)

Vous avez aussi tenté d'installer un cert zcam avec EKU "Remote Desktop Authentication" sur la cible. Si le cert n'inclut PAS vip.zres.lan dans son SAN, le client 25H2 (qui se connecte par le nom de la VIP) rejette le cert. Le cert doit lister TOUS les noms par lesquels les clients peuvent se connecter.

📡
Trace réseau vide entre VIP et cible
Interprétation : l'échec se produit côté client OU dans la phase F5 amont. Si le client coupe la connexion en CredSSP, le F5 referme la connexion backend (ou ne l'ouvre jamais).
🪪
Cert F5 invisible dans l'erreur
Interprétation : en TCP bridge, le cert F5 n'est jamais présenté. L'erreur survient avant/pendant CredSSP. Le client ne sait pas qu'un cert existe sur le F5.
🎫
Cert zcam sur cible : échec aussi
Interprétation : le cert seul ne résout rien si Kerberos/NLA échoue en amont. Le cert n'est validé qu'APRÈS CredSSP réussi. Le problème est en amont.

Diagnostic étape par étape — 11 vérifications ordonnées

Suivez ces vérifications dans l'ordre. Chaque étape vise à isoler la source du problème sans introduire d'effets de bord. Notez les résultats à chaque étape.

  1. Vérifier la résolution DNS de la VIP depuis le client
    Sur le poste 25H2 : nslookup vip.zres.lan puis Resolve-DnsName vip.zres.lan -Type A. La réponse doit donner l'IP de la VIP F5. Si NXDOMAIN, configurer un forwarder DNS conditionnel zcam → zres ou un stub zone.
  2. Tester la connectivité TCP brute vers la VIP
    Test-NetConnection -ComputerName vip.zres.lan -Port 3389. Doit retourner TcpTestSucceeded : True. Si False, c'est un problème réseau/firewall avant le F5.
  3. Vérifier le statut du pool member F5
    Sur la console F5 : tmsh show ltm pool <pool_name> members. Le membre doit être Available (Enabled). Si OFFLINE : revoir le monitor (privilégier tcp_half_open sur 3389 ou un monitor RDP custom).
  4. Capturer le trafic côté F5 (deux interfaces)
    tcpdump -nni 0.0:nnp host <client_ip> -w /shared/tmp/client-side.pcap et idem pour <target_ip>. Comparer : si client-side voit du TLS mais server-side ne voit rien, le F5 ne forwarde pas (pool down OU client coupe trop tôt).
  5. Vérifier la configuration du Virtual Server F5
    tmsh list ltm virtual <vs_name> profiles. Pour TCP bridge : doit lister tcp uniquement. La présence d'un clientssl profile indique que TLS est terminé sur le F5 → à supprimer pour rétablir le mode bridge.
  6. Inspecter les SPN enregistrés pour la cible
    Sur un DC zcam : setspn -L <hostname-cible>. Doit lister au minimum TERMSRV/<hostname> et TERMSRV/<FQDN>.zcam.local. Le SPN TERMSRV/vip.zres.lan est probablement absent — c'est très souvent la cause racine.
  7. Vérifier la présence d'un trust entre zcam et zres
    Sur un DC zcam : nltest /domain_trusts ou Get-ADTrust -Filter *. Si aucun trust avec zres : le referral cross-realm Kerberos est impossible. Toutefois, un SPN local sur le compte machine cible peut contourner ce point.
  8. Lancer une trace Kerberos côté client
    klist purge puis tenter la connexion RDP, puis klist. Si aucun ticket pour termsrv/vip.zres.lan n'apparaît, Kerberos a échoué. Activer le logging Kerberos : HKLM\...\Lsa\Kerberos\Parameters\LogLevel = 1 et consulter Event Viewer (System).
  9. Examiner les Event Logs côté client 25H2
    TerminalServices-ClientActiveXCore/Admin et System. Chercher : Event ID 1024/1026 (RDP), 36874 (SChannel TLS), 4625 (auth). Le code 0x80004005 + "function requested is not supported" = signature CredSSP.
  10. Examiner les Event Logs côté cible 23H2
    Si vide : confirme que la connexion ne l'atteint pas. Si présence d'Event ID 1149 : la connexion a touché la cible mais auth a échoué après → cause différente, à creuser via Event 4625.
  11. Tester en bypass F5 (connexion directe à la cible)
    mstsc /v:<FQDN-cible>.zcam.local. Si ça marche en direct : le problème est lié à la VIP (SPN, cert, F5 config). Si ça échoue aussi : problème intrinsèque 25H2 ↔ 23H2 (durcissement CredSSP).
📊
Tableau d'interprétation des résultats Si étape 2 ✓ mais 4 montre client-side TLS sans server-side → SPN ou F5 pool. Si étape 11 ✓ en direct mais ✗ via VIP → SPN/DNS spécifique à la VIP. Si étape 11 ✗ aussi → durcissement 25H2 incompatible avec cible 23H2 non patchée.
07
Solutions complètes — 8 approches du plus simple au plus structurant

Solutions classées par priorité de mise en œuvre. Combinez plusieurs solutions selon vos contraintes. Chaque solution est testable indépendamment.

1
Enregistrer le SPN TERMSRV/vip.zres.lan sur le compte machine de la cible
FACILE · 5 min RECOMMANDÉ #1

C'est très probablement la solution principale. En ajoutant un SPN supplémentaire à la cible (compte machine dans zcam), Kerberos peut résoudre TERMSRV/vip.zres.lan vers la cible, même si la VIP n'est pas un objet AD. Pas besoin de trust avec zres.

# Sur un DC zcam ou poste admin avec RSAT setspn -S TERMSRV/vip.zres.lan HOSTNAME-CIBLE$ setspn -S TERMSRV/vip HOSTNAME-CIBLE$ # Pour le nom court # Vérifier l'enregistrement setspn -L HOSTNAME-CIBLE$ # Doit lister TERMSRV/vip.zres.lan + TERMSRV/hostname.zcam.local # Côté client, purger les tickets Kerberos pour forcer un re-fetch klist purge klist purge -li 0x3e7 # Inclut le contexte SYSTEM
⚠️
Contrainte Un SPN doit être unique dans la forêt. Si une autre machine a déjà TERMSRV/vip.zres.lan, setspn échoue. Vérifier avec setspn -Q TERMSRV/vip.zres.lan. Si conflit : nettoyer l'ancien avec setspn -D.
2
Déployer un certificat sur la cible avec SAN incluant vip.zres.lan
MOYEN · 30 min RECOMMANDÉ #2

Le certificat sur la cible doit contenir tous les noms d'accès possibles en Subject Alternative Name : le hostname, le FQDN, ET le nom de la VIP (vip.zres.lan). Le cert doit avoir l'EKU "Server Authentication" (1.3.6.1.5.5.7.3.1) et idéalement "Remote Desktop Authentication" (1.3.6.1.4.1.311.54.1.2).

# Sur l'AD CS de zcam : dupliquer le modèle "Computer" → "RDS-VIP-Cert" # Extensions → Application Policies → ajouter "Remote Desktop Authentication" # Demande explicite avec SAN multiple via certreq (sur la cible) # Fichier request.inf : [NewRequest] Subject = "CN=hostname.zcam.local" KeyLength = 2048 MachineKeySet = TRUE RequestType = PKCS10 [Extensions] 2.5.29.17 = "{text}" _continue_ = "DNS=hostname.zcam.local&" _continue_ = "DNS=hostname&" _continue_ = "DNS=vip.zres.lan&" _continue_ = "DNS=vip" 2.5.29.37 = "{text}1.3.6.1.5.5.7.3.1,1.3.6.1.4.1.311.54.1.2" certreq -new request.inf request.req certreq -submit -config "ca.zcam.local\zcam-CA" request.req cert.cer certreq -accept cert.cer # Lier le cert au service RDP $thumb = (Get-ChildItem Cert:\LocalMachine\My | ? {$_.Subject -like "*hostname*"}).Thumbprint Set-WmiInstance -Path "\\.\root\cimv2\TerminalServices:Win32_TSGeneralSetting.TerminalName='RDP-Tcp'" -Arguments @{SSLCertificateSHA1Hash=$thumb} # Push GPO : Remote Desktop Session Host → Security # → "Server authentication certificate template" = RDS-VIP-Cert
3
Établir une relation d'approbation zcam ↔ zres (ou stub zone DNS)
DIFFICILE · plusieurs jours STRUCTURANT

Si beaucoup de ressources zres doivent être accessibles depuis zcam, un trust externe ou de forêt apporte une solution durable. Mais pour résoudre uniquement votre problème RDP via VIP, la Solution 1 est suffisante et nettement plus rapide.

# Création d'un external trust depuis un DC zcam netdom trust zcam.local /Domain:zres.local /Add /UserD:Administrator /PasswordD:* /UserO:Administrator /PasswordO:* # Vérifier le trust nltest /domain_trusts /v Get-ADTrust -Filter * # Forwarder DNS conditionnel (alternative légère) Add-DnsServerConditionalForwarderZone -Name "zres.local" -ReplicationScope "Forest" -MasterServers 10.x.x.x
4
Vérifier & corriger la configuration F5 (TCP profile only)
FACILE · 15 min CRITIQUE

S'assurer que la VIP est en TCP bridge pur (pas de SSL termination). Si un Client SSL profile traîne, le supprimer. Vérifier que le monitor pool est tcp ou un monitor RDP custom.

# Vérifier le Virtual Server tmsh list ltm virtual RDP-VIP-3389 all-properties # Profiles attendus : tcp uniquement. PAS de clientssl ni serverssl. # Si clientssl présent → le retirer : tmsh modify ltm virtual RDP-VIP-3389 profiles delete { clientssl } # Vérifier le pool member (doit être Available/Enabled) tmsh show ltm pool RDP-Pool members # Monitor recommandé pour RDP tmsh create ltm monitor tcp rdp-3389-mon { destination *:3389 interval 10 timeout 31 } tmsh modify ltm pool RDP-Pool monitor rdp-3389-mon # SNAT — important si client et backend dans subnets différents tmsh modify ltm virtual RDP-VIP-3389 source-address-translation { type automap } # Vérifier connectivité depuis F5 vers backend tmsh run util tcpdump -nni 0.0:nnp host <target_ip> and port 3389
5
Désactiver temporairement NLA sur la cible (DIAGNOSTIC uniquement)
FACILE · 2 min DIAGNOSTIC

À utiliser uniquement pour valider si NLA est la cause. Ne pas garder en production : NLA est une protection essentielle contre les attaques pre-auth.

# Sur la cible 23H2 (PowerShell admin) Set-ItemProperty -Path 'HKLM:\SYSTEM\CurrentControlSet\Control\Terminal Server\WinStations\RDP-Tcp' -Name "UserAuthentication" -Value 0 Restart-Service TermService -Force # Tester la connexion. Si ça marche → CredSSP/NLA est le bloqueur. # IMPORTANT : RÉACTIVER NLA APRÈS LE TEST (-Value 1)
6
Créer un alias DNS dans zcam pointant vers la VIP F5
FACILE · 10 min CONTOURNEMENT ÉLÉGANT

Au lieu de demander aux clients de se connecter à vip.zres.lan, créer un enregistrement DNS dans zcam (ex : rdp-prod.zcam.local) qui résout vers l'IP de la VIP F5. Ainsi le client cherche le SPN dans son propre domaine (zcam), ce qui évite le cross-realm.

# Sur DNS zcam, créer un A record Add-DnsServerResourceRecordA -Name "rdp-prod" -ZoneName "zcam.local" -IPv4Address "10.zres.x.x" # Côté cible 23H2 — ajouter le SPN correspondant setspn -S TERMSRV/rdp-prod.zcam.local HOSTNAME-CIBLE$ # Push un cert avec SAN incluant rdp-prod.zcam.local (voir Solution 2) # Le client se connecte désormais ainsi : mstsc /v:rdp-prod.zcam.local
✨
Pourquoi cette solution est élégante Tout reste dans zcam : SPN, certificat, DNS, identité. Aucun trust nécessaire. Le F5 reste en zres mais c'est transparent — c'est juste une IP que le client atteint via un nom zcam.
7
Assouplir temporairement CredSSP côté client (TEST uniquement)
FACILE · 2 min DIAGNOSTIC

Permettre temporairement un fallback CredSSP "Mitigated" pour vérifier si le durcissement Protected Mode est le bloqueur. Ne jamais conserver en production.

# Sur le client 25H2 (test diagnostic) Set-ItemProperty -Path 'HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\Policies\System\CredSSP\Parameters' -Name "AllowEncryptionOracle" -Value 2 # 0 = Force Updated (défaut 25H2) | 1 = Mitigated | 2 = Vulnerable (DANGEREUX) # Restaurer le défaut IMMÉDIATEMENT après le test Remove-ItemProperty -Path 'HKLM:\SOFTWARE\...\CredSSP\Parameters' -Name "AllowEncryptionOracle"
8
Migrer vers un RD Gateway (architecture cible long terme)
STRUCTURANT · projet ARCHITECTURE

RD Gateway termine RDP sur HTTPS (port 443), authentifie les utilisateurs avec MFA possible, et établit ensuite une connexion RDP interne vers la cible. Remplace avantageusement le F5 sur ce flux et règle naturellement les problèmes de SPN/cert.

  • Avantages : terminaison TLS sur 443 propre, cert unique valide, NLA fonctionnel, NPS policies, journalisation centralisée.
  • F5 : peut rester en load-balancing devant deux RD Gateway HA en SSL Bridge mode (cert F5 valide en frontal).
  • Cible : reste inchangée. Le RD Gateway parle RDP standard avec elle.

TLS Alert codes
Alert 40 (handshake_failure) = mauvaise version TLS. Alert 42 (bad_certificate) = chaîne invalide. Alert 46 (unsupported_certificate) = cert pas reconnu (EKU manquant). Alert 71 (insufficient_security) = cipher trop faible.
🎫
Kerberos — codes d'erreur
KRB-ERROR code 7 = S_PRINCIPAL_UNKNOWN (SPN absent). Code 25 = PREAUTH_REQUIRED (normal). Code 24 = PREAUTH_FAILED (mauvais mdp). Code 18 = CLIENT_REVOKED (compte désactivé).
⚡
CredSSP — signature
Après TLS handshake réussi, recherche le pattern Negotiate dans le payload chiffré. Si le client ferme la session avec FIN après le 1er TLS Application Data, CredSSP a échoué.
