Synthèse exécutive

Ce qui se passe et pourquoi

⚠

Le problème

Windows 11 25H2 valide strictement le certificat TLS présenté lors de la phase NLA. Le F5 en SSL Bridging présente son propre certificat au client — sans l'EKU Remote Desktop Authentication obligatoire depuis 25H2.

⬡

Le mécanisme

Le F5 crée deux sessions TLS distinctes : une côté client (avec le cert F5), une côté VDI (avec le cert VDI). Le client 25H2 ne voit jamais le cert VDI — il voit celui du F5, le rejette, et bloque la connexion avant toute saisie d'identifiants.

✓

La solution

Passer le Virtual Server F5 en mode SSL Passthrough (Performance Layer 4 / FastL4). Le client voit directement le certificat de la VDI. Aucune modification côté VDI ou PKI requise.

02 — Analyse de la cause racine

Pourquoi Windows 11 25H2 refuse la connexion

La mise à jour 25H2 (avril 2026) transforme la validation du certificat TLS de la phase NLA en blocage dur. Ce qui était un avertissement ignorable est devenu une erreur fatale et non contournable côté client.

Le comportement du F5 en SSL Bridging

En mode SSL Bridging, le F5 termine la session TLS initiale du client puis réinitie une nouvelle session TLS vers le backend. Il agit comme un proxy TLS intermédiaire. Conséquence directe : c'est le certificat du F5 — et non celui de la VDI — qui est présenté au client lors de la phase NLA.

03 — Notion de "Trusted"

Ce que signifie un certificat "trusted" et par qui

Un certificat est trusted exclusivement du point de vue du client — la machine Windows 11 25H2. Windows maintient un magasin de certificats racines appelé Trusted Root Certification Authorities. Lors de la validation TLS, le client remonte la chaîne de certification jusqu'à la CA racine et vérifie sa présence dans ce magasin. Si la CA racine n'est pas dans le magasin : erreur, connexion refusée. Indépendamment de tout autre critère.

En homologation — ygie domain-joined

Le poste ygie est rattaché à votre domaine AD. Il reçoit automatiquement via GPO Active Directory les certificats racines de votre PKI interne.

Le certificat de la VDI ycam, émis par cette PKI, est donc trusté par ce client. En passthrough, la connexion passe.

En production — clients externes non-gérés

Les postes clients non-gérés par votre IT n'ont pas votre PKI interne dans leur Trusted Root store. Même avec le passthrough actif, si votre CA interne est inconnue du client, le certificat VDI déclenchera une erreur "CA non reconnue".

Ce point doit être qualifié séparément.

Point de vigilance prod : Le passthrough résout le problème d'EKU absent sur le cert F5. Mais si les clients externes ne font pas confiance à votre PKI interne, ils auront une erreur distincte (CA non trustée) même avec le cert VDI correct. Ces deux problèmes sont indépendants et doivent être traités séparément.

04 — Architecture F5

Comment fonctionne une VIP F5

Un Virtual Server (VS) est l'objet central du F5 BIG-IP. Il écoute sur une adresse IP virtuelle (la VIP) et un port définis, puis applique des profils et politiques au trafic avant de le router vers un pool de serveurs backend. C'est l'ensemble de ces profils qui détermine si le F5 agit en proxy TLS (Bridging) ou en relais transparent (Passthrough).

Anatomie d'un Virtual Server F5 BIG-IP — composants et rôles
Composant Rôle Impact RDP
Virtual Server Point d'entrée. IP VIP + port. Orchestre tous les autres objets et définit le mode de traitement. Détermine si le trafic est traité en L7 (Bridging) ou L4 (Passthrough).
Client SSL Profile Termine la session TLS côté client. C'est lui qui présente le certificat F5 au client RDP. Cause directe du problème. À supprimer en passthrough.
Server SSL Profile Établit une nouvelle session TLS vers le backend VDI. Opaque pour le client. Invisible du client. À supprimer en passthrough.
FastL4 Profile Forwarding Layer 4 accéléré via ASIC hardware. Le F5 ne déchiffre pas, il route les octets. C'est ce profil qui active le passthrough transparent.
Default Pool Groupe de VDI backend avec health monitor et algorithme de load balancing. Inchangé entre Bridging et Passthrough.
05 — Mode passthrough

Comment fonctionne le SSL Passthrough

En mode SSL Passthrough (Performance Layer 4 / FastL4), le F5 n'inspecte pas le contenu TLS. Il agit comme un aiguilleur TCP transparent : il reçoit le SYN client, sélectionne un membre du pool backend selon la règle de load balancing, puis "splice" les deux connexions TCP. Chaque octet reçu d'un côté est transmis tel quel à l'autre côté. La handshake TLS entre le client et la VDI passe intégralement sans interception.

07 — Configuration exacte

Mise en œuvre du SSL Passthrough sur F5 BIG-IP

Prérequis : Cette configuration s'applique au Virtual Server existant qui porte la VIP RDP vers les VDI. Le pool et ses membres restent strictement inchangés. Un change doit être ouvert et la modification effectuée en fenêtre de maintenance.

1

Modifier le type du Virtual Server

Changer le type de Standard à Performance (Layer 4). C'est la modification fondamentale qui active le mode passthrough transparent.

TMSH — F5 BIG-IP
tmsh modify ltm virtual <nom-du-vs> \
    ip-protocol tcp \
    profiles replace-all-with { fastL4 { } }
2

Supprimer les profils SSL Client et Server

Retirer le Client SSL Profile et le Server SSL Profile du Virtual Server. En mode FastL4, ces profils sont incompatibles et doivent être absents.

Interface GUI — Local Traffic › Virtual Servers › [NomVS]
Onglet Properties :
  Type                        : Performance (Layer 4)

Onglet SSL Profiles (Client) :
  Selected profiles           : [vide — retirer tous les profils]

Onglet SSL Profiles (Server) :
  Selected profiles           : [vide — retirer tous les profils]
3

Appliquer le profil FastL4

Le profil fastL4 est le profil système F5 qui active le forwarding matériel Layer 4 via ASIC. Il remplace le profil TCP standard.

TMSH
tmsh modify ltm virtual <nom-du-vs> \
    profiles add { fastL4 { context all } }

# Vérification de la configuration résultante
tmsh show ltm virtual <nom-du-vs> detail
4

Configurer la persistence Source Address Affinity (recommandé)

En passthrough L4, la persistence ne peut pas être basée sur le cookie (L7). Utiliser la Source Address Affinity pour s'assurer que les reconnexions d'un même client vont sur la même VDI — essentiel pour la continuité des sessions VDI.

TMSH — Création et application du profil de persistence
tmsh create ltm persistence source-addr rdp-src-affinity {
    timeout 3600
    match-across-services enabled
    match-across-virtuals disabled
}

tmsh modify ltm virtual <nom-du-vs> {
    persist replace-all-with {
        rdp-src-affinity { default yes }
    }
}
5

Vérifier le health monitor du pool

S'assurer que le pool utilise un health monitor TCP sur le port 3389 uniquement. Un monitor HTTPS ou SSL tenterait une handshake TLS depuis le F5 vers les VDI, ce qui peut perturber le passthrough.

TMSH — Vérification et création du monitor TCP
tmsh show ltm pool <nom-du-pool> members detail

# Si le monitor actuel est HTTPS/SSL, le remplacer :
tmsh create ltm monitor tcp rdp-tcp-monitor {
    interval 10
    timeout  31
    dest     *.3389
}

tmsh modify ltm pool <nom-du-pool> {
    monitor rdp-tcp-monitor
}
6

Sauvegarder et valider

TMSH — Sauvegarde de la configuration
tmsh save sys config

# Test de connexion depuis un poste 25H2
# Le certificat présenté doit désormais être celui de la VDI,
# et non plus celui du F5.

08 — Plan d'action

Recommandations & roadmap

Point critique production : Le passthrough résout le problème d'EKU manquant. Mais si les clients externes de prod ne font pas confiance à votre PKI interne (CA absente de leur Trusted Root store), ils recevront une erreur "CA non reconnue" distincte. Ce point doit être qualifié avant tout déploiement en prod.

Court terme — J+0 à J+5

Déblocage homologation

  • Ouvrir un change F5 pour le VS d'homologation
  • Passer le VS en Performance L4 / FastL4
  • Supprimer Client SSL et Server SSL profiles
  • Valider la connexion depuis le poste 25H2 ygie
  • Optionnel : GPO workaround en urgence si blocage utilisateurs critique
Moyen terme — J+5 à J+30

Qualification production

  • Qualifier la confiance PKI des clients externes prod
  • Si non trustée : évaluer CA croisée ou certificat issu d'une racine Windows-native
  • Préparer le change F5 pour le VS de prod (zcam)
  • Tester la persistence Source Address Affinity sur les VDI
  • Valider le health monitor TCP:3389
Long terme — J+30+

Pérennisation

  • Déployer sur l'ensemble des VS RDP de production
  • Documenter l'architecture passthrough dans la CMDB
  • Mettre à jour les runbooks d'exploitation F5
  • Intégrer dans les critères d'acceptation des futures mises à jour Windows
  • Revue PKI globale si clients multi-domaines non-managés
