Proposition : compte de service partagé (gMSA) pour RDP Kerberos

VIp-coop.zres.ztech · F5 L4 · pool de 5 RDSH · Kerberos end-to-end
❌ non applicable à TermService

L'architecte propose de créer un compte de service partagé (gMSA), d'y poser le SPN de la VIP, et de faire tourner TermService sous ce compte sur chaque backend — afin que tous partagent une même clé Kerberos. Le raisonnement est exactement le bon modèle pour les services en ferme (IIS, SharePoint, SQL). Le problème est que TermService est le seul service Windows courant où ce levier n'existe pas.

Pourquoi c'est le bon réflexe — en général

Le modèle compte de service partagé + SPN sur ce compte est la recette officielle de Microsoft pour Kerberos derrière un load-balancer. Le SPN doit être unique → il doit vivre sur un seul compte → tous les nœuds doivent tourner sous ce compte pour partager la clé de déchiffrement. C'est comme ça qu'on authentifie en Kerberos une ferme IIS, SharePoint, SQL Server…

Ferme IIS / SharePoint

AppPool = svc-app sur tous les nœuds
Kerberos : 3/3 ✓

Ferme RDP (TermService)

TermService = NetworkService (identité machine)
Kerberos : 0/3 ✗ → NTLM
La différence critique : dans IIS, l'identité du pool d'applications est un réglage de première classe (modifiable, supporté) → le processus accepte les AP-REQ avec la clé du compte de service. Dans RDP, l'authentification NLA/CredSSP est ancrée à l'identité machine (NetworkService + clés privées RSA machine) → le backend tente de déchiffrer l'AP-REQ avec sa clé machine, pas celle du gMSA, quel que soit le compte du service.
Pourquoi TermService ne peut pas utiliser un gMSA
Architecture NLA

L'authentification RDP est dans le contexte machine

Le listener RDP-Tcp accepte les connexions et traite NLA/CredSSP dans le contexte de la machine (NetworkService → LSASS → clé du compte ordinateur). L'AP-REQ est déchiffré avec la clé machine, pas celle du service configuré dans la console Services.

Dépendances machine

TermService dépend des clés privées RSA de la machine

Le certificat TLS du listener, les MachineKeys dans Crypto\RSA, les privilèges de création de session — tout est lié à NetworkService et à l'identité machine. Changer le compte du service casse cet accès.

Non supporté

Microsoft ne propose pas ce réglage

Contrairement à IIS (AppPool identity) ou SQL Server (service account), il n'existe aucun réglage supporté pour dire à TermService d'accepter un ticket avec une autre clé que celle de la machine. La réponse Microsoft pour les fermes RDS est le RD Connection Broker.

Ce qui se passe avec vos 5 backends

Même si le gMSA est installé, même si le SPN est en place — le ticket est scellé avec la clé de svc-vip$ (sarcelle), et chaque backend tente de l'ouvrir avec sa propre clé machine. Aucune ne correspond.

Le KDC scelle le ticket avec la clé du compte svc-vip$ (cadenas sarcelle). Chaque backend, en recevant l'AP-REQ dans le tunnel TLS, tente de l'ouvrir avec sa propre clé machine (verte, bleue, ambre, rouge, violette). Aucune ne correspond → KRB_AP_ERR_MODIFIED → repli NTLM sur les 5. Résultat : 0/5 en Kerberos, exactement comme si vous n'aviez rien fait — le gMSA est installé mais n'est jamais sollicité par TermService.
sarcelle = gMSA svc-vip$ (la clé du cadenas) A B C D E = clés machine des 5 backends

Règle : un cadenas a la couleur de la clé qui l'a fermé, et il ne s'ouvre qu'avec une clé de la même couleur. Le gMSA (sarcelle) n'est utilisé par aucun processus côté backend → sa clé reste dans AD, jamais chargée en mémoire.

Nuance — où ce modèle marcherait pour RDP
FreeRDP (Linux). Si le serveur RDP n'est pas le TermService Windows mais FreeRDP (proxy ou shadow server sous Linux), vous contrôlez le keytab et pouvez authentifier le service avec TERMSRV/VIp-coop@ZRES.ZTECH posé sur le compte de service — ça fonctionne en NLA + Kerberos. Mais ça implique de remplacer la pile RDP Windows par une stack que vous gérez vous-même, ce qui n'est généralement pas envisageable dans un environnement RDSH de production.
Les alternatives supportées pour du Kerberos en ferme RDP
Option A

RD Connection Broker

Le broker redirige le client vers l'hôte de session réel. Kerberos vise alors TERMSRV/<vrai-hôte> sur le compte machine de cet hôte — qu'il possède forcément. C'est la réponse Microsoft supportée. Le client reçoit le nom de l'hôte cible et refait un nouveau TGS-REQ ciblé.

Option B

F5 APM + KCD (S4U2Proxy)

Le F5 termine CredSSP, pré-authentifie l'utilisateur, puis obtient un ticket vers le vrai SPN de chaque backend au nom de l'utilisateur (Kerberos Constrained Delegation). Kerberos de bout en bout — au prix du L4 pur et du CredSSP natif.

Option C

Un seul backend

Un seul RDSH derrière la VIP, avec TERMSRV/VIp-coop sur le compte machine de cet hôte unique : une seule clé → ça marche toujours. Mais zéro répartition de charge, zéro haute disponibilité.

« Le bon réflexe, le mauvais service. »
Le modèle compte-de-service + SPN partagé est exactement le bon — c'est juste que TermService est la seule brique courante où ce levier n'existe pas, contrairement à IIS. Pour du RDP en ferme, c'est RD Connection Broker ou APM + KCD.
