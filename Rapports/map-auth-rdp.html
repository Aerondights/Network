Conditions de bon fonctionnement — NTLMv2

une fois NTLM autorisé · à travers la VIP F5 (L4)
1CredSSP / NLA laisse arriver jusqu'à NTLM
  • Identité TLS valide : le certificat de MachineDestination couvre VIp-coop.zres.ztech dans son SAN (+ EKU Remote Desktop Authentication). Déjà en place.
  • Kerberos échoue proprement : aucun SPN TERMSRV/VIp-coop ni HOST/VIp-coop sur un compte → SSPI bascule.
  • Patch CredSSP aligné des deux côtés (Encryption Oracle Remediation, CVE-2018-0886).
2NTLMv2 accepté des deux côtés
  • LmCompatibilityLevel ≥ 3 (idéal 5) sur le client et MachineDestination.
  • L'exception Restrict NTLM vise le nom de connexion VIp-coop.zres.ztech, pas le nom de l'hôte cible.
  • Sécurité de session compatible (128 bits + NTLMv2 session security) — défaut sur Win11.
3La validation des identifiants peut se faire
  • MachineDestination jointe au domaine, canal sécurisé sain (Test-ComputerSecureChannel).
  • Un DC joignable et réactif pour le domaine de l'utilisateur (chemin d'approbation OK en multi-domaine).
  • Compte utilisateur valide : activé, non verrouillé, mot de passe correct, heures/stations autorisées.
4Spécifique au relais L4 / F5
  • Le F5 reste en pur L4 (pas de terminaison/bridging TLS) — sinon CredSSP casse (clé publique différente).
  • Tout le trio NEGOTIATE / CHALLENGE / AUTHENTICATE reste sur le même backend : persistance source-address + idle timeout ≥ durée de session.
Non requis (contrairement à Kerberos) : aucun SPN, le nom VIp-coop n'entre jamais dans le calcul cryptographique, et la synchronisation d'horloge n'est pas bloquante.
