00 — Contexte préalable

Attention : deux problèmes distincts coexistent

Avant tout, il est crucial de comprendre que l'environnement Windows 11 25H2 + mise à jour d'avril 2026 (KB5083769) génère deux problèmes indépendants qui peuvent sembler identiques mais ont des causes racines, des mécanismes et des solutions totalement différentes.

Erreur fréquente : Confondre ces deux problèmes conduit à appliquer la mauvaise solution. Une solution qui corrige le problème B n'aura aucun effet sur le problème A, et vice versa. La vidéo YouTube analysée ici traite exclusivement du problème B.

Problème A — Le sujet de votre rapport

⛔ Échec NLA / TLS — EKU manquant

Le client 25H2 refuse de s'authentifier via NLA car le certificat présenté par le F5 en SSL Bridging ne contient pas l'EKU "Remote Desktop Authentication" (OID 1.3.6.1.4.1.311.54.1.2). La connexion échoue avant même l'affichage de la fenêtre de saisie d'identifiants.

  • Se produit lors de la phase NLA (Network Level Authentication)
  • Erreur : "Erreur d'authentification" / certificat invalide
  • Cause : F5 en SSL Bridging interpose son propre cert
  • Apparu avec Windows 11 25H2 (validation EKU devenue bloquante)
Problème B — Le sujet de la vidéo YouTube

⚠ Avertissement à l'ouverture d'un fichier .rdp

Après KB5083769 (Patch Tuesday avril 2026), l'ouverture d'un fichier .rdp non signé déclenche un avertissement de sécurité répété et désactive par défaut la redirection des ressources locales (presse-papiers, imprimantes, disques).

  • Se produit à l'ouverture du fichier .rdp lui-même
  • Erreur : dialogue de sécurité "risque de phishing potentiel"
  • Cause : le fichier .rdp n'est pas signé numériquement
  • Apparu avec KB5083769 — avril 2026
Chronologie — Deux problèmes introduits à des moments différents
01 — Analyse vidéo YouTube

Que propose la vidéo woshub.com ? (Problème B uniquement)

La vidéo YouTube (https://www.youtube.com/watch?v=AykyiZcbZac) — publiée le 19 avril 2026 par le canal Windows OS Hub (woshub.com) — s'intitule : "Remove the Security Warning When Opening an RDP File in Windows 11 After April 2026 Update". Elle répond au Problème B uniquement : les avertissements de sécurité à l'ouverture de fichiers .rdp introduits par KB5083769.

Deux solutions sont présentées dans la vidéo :

Solution YT-1 — Clé de registre (workaround rapide)

Désactiver le dialogue d'avertissement en forçant l'ancien comportement via une valeur de registre. Comme donner un passe-droit à la fenêtre qui crie "attention !" — elle sera bâillonnée.

CMD ou PowerShell (Admin) — Clé de registre workaround Workaround temporaire 
# Méthode CMD (rapide, déployable par GPO)
REG ADD "HKLM\Software\Policies\Microsoft\Windows NT\Terminal Services\Client" /v "RedirectionWarningDialogVersion" /t REG_DWORD /d "1" /F

# Méthode PowerShell équivalente
Set-ItemProperty -Path "HKLM:\Software\Policies\Microsoft\Windows NT\Terminal Services\Client" `
    -Name "RedirectionWarningDialogVersion" -Type DWord -Value 1 -Force

# Vérification
Get-ItemProperty -Path "HKLM:\Software\Policies\Microsoft\Windows NT\Terminal Services\Client" `
    -Name "RedirectionWarningDialogVersion"

Solution YT-2 — Signer le fichier .rdp avec rdpsign.exe (solution recommandée par la vidéo)

Apposer une signature numérique sur le fichier .rdp, comme un notaire appose son cachet sur un document officiel. Windows vérifie le cachet : si le cachet vient d'une autorité en laquelle il a confiance, plus d'avertissement.

Cela nécessite un certificat avec l'EKU Code Signing (OID 1.3.6.1.5.5.7.3.3) — à ne pas confondre avec l'EKU Remote Desktop Authentication (OID 1.3.6.1.4.1.311.54.1.2) du Problème A.

PowerShell (Admin) — Création certificat auto-signé + signature du fichier .rdp Solution recommandée pour Problème B 
# ── ÉTAPE 1 : Créer un certificat Code Signing (si pas de PKI d'entreprise)
$cert = New-SelfSignedCertificate `
    -Subject "CN=RDP Publisher Corp" `
    -Type CodeSigningCert `
    -CertStoreLocation cert:\LocalMachine\My `
    -KeyUsage DigitalSignature `
    -NotAfter (Get-Date).AddYears(3)

# Récupérer l'empreinte SHA256
$thumbprint = $cert.Thumbprint
Write-Host "Thumbprint : $thumbprint" -ForegroundColor Cyan

# ── ÉTAPE 2 : Signer le fichier .rdp
# (depuis la machine où la clé privée est disponible — poste admin)
rdpsign.exe /v /sha256 $thumbprint "C:\RDP\vdi-prod.rdp"

# Réponse attendue si succès :
# Successfully signed: C:\RDP\vdi-prod.rdp

# ── ÉTAPE 3 : Exporter le certificat public (sans clé privée)
Export-Certificate -Cert "Cert:\LocalMachine\My\$thumbprint" `
    -FilePath "C:\RDP\rdp-publisher.cer"

# ── ÉTAPE 4 : Distribuer aux clients via GPO (sur les machines cibles)
# Importer dans Trusted Publishers + Trusted Root via GPO ou manuellement :
Import-Certificate -FilePath "C:\RDP\rdp-publisher.cer" `
    -CertStoreLocation Cert:\LocalMachine\TrustedPublisher
Import-Certificate -FilePath "C:\RDP\rdp-publisher.cer" `
    -CertStoreLocation Cert:\LocalMachine\Root

# ── ÉTAPE 5 : Configurer GPO pour reconnaître le publisher (optionnel)
# Computer Config → Admin Templates → Windows Components
# → Remote Desktop Services → RDP Connection Client
# → "Specify SHA1 thumbprints of certificates representing trusted .rdp publishers"
# Valeur : l'empreinte SHA1 (pas SHA256) du certificat, sans espaces
$thumbprintSHA1 = (Get-Item "Cert:\LocalMachine\My\$thumbprint").GetCertHashString()
Write-Host "SHA1 pour GPO : $thumbprintSHA1"

02 — Verdict de viabilité

Les techniques de la vidéo sont-elles viables pour votre problème ?

Réponse courte : Non pour le Problème A (NLA/EKU), Oui pour le Problème B (avertissement .rdp). Voici pourquoi en détail :

Solution YT-1 — Clé registre RedirectionWarningDialogVersion

✗ NON VIABLE pour le Problème A (votre incident)

Cette clé contrôle uniquement l'affichage du dialogue de sécurité à l'ouverture d'un fichier .rdp. Elle n'a aucun effet sur la validation du certificat TLS lors de la phase NLA. Même avec cette clé à 1, un client 25H2 connecté via la VIP F5 (SSL Bridging) continuera d'échouer à la NLA car le certificat du F5 ne contient pas l'EKU Remote Desktop Authentication (OID 1.3.6.1.4.1.311.54.1.2). Ces deux mécanismes sont dans des couches logicielles totalement séparées.

◑ VIABLE PARTIELLEMENT pour le Problème B (complémentaire)

Comme workaround temporaire pour supprimer le dialogue de sécurité à l'ouverture des .rdp. Déployable en urgence par GPO. Non recommandé à long terme car il réintroduit le comportement que Microsoft a intentionnellement sécurisé avec KB5083769.

Solution YT-2 — Signature .rdp avec rdpsign.exe

✗ NON VIABLE pour le Problème A (votre incident)

La signature d'un fichier .rdp n'affecte pas le certificat présenté par le serveur lors de la connexion TLS. Le fichier .rdp est signé avec un certificat Code Signing (EKU OID 1.3.6.1.5.5.7.3.3). Ce certificat est vérifié avant l'établissement de la connexion TCP. La NLA, elle, vérifie un tout autre certificat — celui que le serveur (ici, le F5) présente pendant le handshake TLS. Ce sont deux objets cryptographiques, deux vérifications, deux EKU et deux moments complètement différents dans le flux de connexion.

✓ VIABLE pour le Problème B (recommandé comme solution permanente)

Solution pérenne pour éliminer les avertissements à l'ouverture des fichiers .rdp et restaurer la redirection des ressources. À combiner avec la solution F5 passthrough (Problème A) pour une résolution complète. Nécessite une PKI d'entreprise ou un certificat Code Signing commercial. La PKI interne AD CS est idéale : tous les postes domain-joinés font confiance automatiquement.

03 — Solutions production disponibles

Quatre options pour corriger le Problème A en production

Voici les quatre options réalistes pour résoudre définitivement le blocage NLA/EKU en production. Elles sont classées par recommandation décroissante pour un environnement VDI avec F5 BIG-IP.

01
★ Recommandée Pérenne Non-invasive VDI Performance accrue
F5 BIG-IP — SSL Passthrough (Performance Layer 4 / FastL4)

Le principe en une phrase : Au lieu que le F5 ouvre la lettre et en écrive une nouvelle (SSL Bridging), le F5 livre la lettre fermée directement au destinataire (SSL Passthrough). Le client voit le vrai certificat de la VDI, avec le bon EKU.

    Avantages
  • Résout définitivement l'EKU manquant sans aucune modification VDI
  • Meilleures performances réseau (FastL4 = accélération hardware ASIC)
  • Sécurité renforcée : le F5 ne déchiffre plus le flux RDP (pas de MITM interne)
  • Compatible 25H2 nativement et durablement — résiste aux prochaines mises à jour Windows
  • Change F5 limité et réversible en quelques minutes
  • Aucune modification côté PKI, VDI, domaine Active Directory
    Contraintes
  • Perte de visibilité L7 sur le flux RDP (le F5 ne déchiffre plus)
  • Persistence uniquement Source Address Affinity (L4) — pas de cookie ou header RDP
  • Clients externes non-managés : la CA interne doit être trustée séparément
  • Ouverture d'un change F5 + fenêtre de maintenance requise
02
Complexe Clients managés uniquement PKI requise
Émettre un certificat avec EKU Remote Desktop Authentication (OID 1.3.6.1.4.1.311.54.1.2) sur le F5

Le principe en une phrase : Donner au F5 un certificat "accrédité RDP" — comme un douanier qui obtient le bon tampon sur son badge. Le F5 continue d'ouvrir les lettres (SSL Bridging), mais son badge est désormais reconnu par 25H2.

    Avantages
  • Conserve la visibilité L7 du F5 (inspection du flux RDP possible)
  • Conserve la persistence L7 (cookie, header)
  • Pas de changement de type de VS sur le F5
  • Architecture actuelle préservée, seul le certificat change
    Contraintes
  • Nécessite un template PKI AD CS avec l'OID 1.3.6.1.4.1.311.54.1.2 ajouté manuellement
  • Les CA publiques commerciales (DigiCert, Sectigo...) n'émettent PAS cet EKU
  • La CA émettrice doit être trustée par TOUS les clients (GPO AD pour les managés)
  • Inapplicable aux clients externes non-managés sans déploiement de CA
  • Configuration PKI complexe, risque d'erreur sur le template
03
Complémentaire Problème B Solution durable
Signature des fichiers .rdp avec rdpsign.exe (solution Problème B — à combiner avec l'option 1)

À utiliser en complément de l'option 1, pas à la place. Résout les avertissements à l'ouverture des fichiers .rdp introduits par KB5083769. Nécessite un certificat Code Signing (OID 1.3.6.1.5.5.7.3.3) émis par votre PKI d'entreprise (AD CS idéalement).

    Avantages
  • Supprime définitivement les dialogues de sécurité à l'ouverture des .rdp
  • Restaure la redirection des ressources (clipboard, imprimantes, disques) par défaut
  • Solution pérenne et alignée avec la politique Microsoft
  • Compatible avec GPO pour distribuer la confiance aux postes domain-joinés
    Contraintes
  • Ne résout PAS le Problème A (NLA/EKU sur F5)
  • Signature doit être réalisée sur la machine avec la clé privée
  • Redistribution des .rdp signés à tous les utilisateurs
  • PKI d'entreprise avec template Code Signing recommandé
04
Workaround court terme Risque sécurité Déconseillé en prod bancaire
GPO — Assouplissement de la validation NLA côté client 25H2

Le principe en une phrase : Dire à Windows "si l'authentification échoue, affiche un avertissement mais laisse passer quand même". Comme lever une barrière routière parce qu'on n'a pas le badge — ça fonctionne, mais c'est une faille de sécurité.

    Avantages
  • Déploiement immédiat sans toucher au F5 ni à la PKI
  • Réversible instantanément via GPO
  • Permet de débloquer les utilisateurs en urgence, le temps du change F5
    Contraintes
  • Réintroduit la vulnérabilité que 25H2 corrige intentionnellement
  • Incompatible avec les audits de sécurité en environnement bancaire / sensible
  • Applicable uniquement aux postes managés domain-joinés (GPO AD)
  • Inapplicable aux clients externes non-managés
  • Doit être retiré impérativement après le change F5
08 — Tests & commandes exactes

Procédures de test détaillées avec commandes précises

Environnement de test présupposé : Toutes les commandes ci-dessous s'appliquent à l'architecture décrite dans votre rapport — Client Windows 11 25H2 (domaine ygie), VIP F5 yres, VDI Windows (domaine ycam). Les noms de variables sont explicitement identifiés pour être remplacés.

Test A — Audit du certificat actuel présenté par le F5 (avant le change)

Avant tout changement, capturer et vérifier l'EKU du certificat présenté par le F5. Comme prendre une photo du problème avant de le réparer.

A1

Capturer le certificat TLS présenté par la VIP F5

Depuis le poste client 25H2 (ou tout poste avec OpenSSL), se connecter en TLS sur le port 3389 de la VIP et extraire le certificat présenté.

PowerShell — Client Windows (25H2 ygie) Audit pré-change 
# ── Test A1 : Capturer le certificat TLS du F5 via PowerShell
$vip = "10.x.x.x"     # ← Remplacer par l'IP de la VIP yres
$port = 3389

$tcpClient = New-Object System.Net.Sockets.TcpClient($vip, $port)
$sslStream = New-Object System.Net.Security.SslStream($tcpClient.GetStream(), $false, {$true})
$sslStream.AuthenticateAsClient($vip)
$cert = $sslStream.RemoteCertificate
$cert2 = New-Object System.Security.Cryptography.X509Certificates.X509Certificate2($cert)

# Afficher le Subject et l'EKU
Write-Host "Subject    : $($cert2.Subject)" -ForegroundColor Cyan
Write-Host "Issuer     : $($cert2.Issuer)" -ForegroundColor Cyan
Write-Host "Thumbprint : $($cert2.Thumbprint)" -ForegroundColor Cyan
Write-Host "Valid until: $($cert2.NotAfter)" -ForegroundColor Cyan
Write-Host ""
Write-Host "Enhanced Key Usages (EKU) :" -ForegroundColor Yellow
$cert2.Extensions | Where-Object {$_.Oid.FriendlyName -eq "Enhanced Key Usage"} | ForEach-Object {
    $eku = New-Object System.Security.Cryptography.X509Certificates.X509EnhancedKeyUsageExtension($_, $false)
    foreach ($oid in $eku.EnhancedKeyUsages) {
        Write-Host "  → $($oid.FriendlyName) ($($oid.Value))" -ForegroundColor White
    }
}

$tcpClient.Close()

# ── RÉSULTAT ATTENDU AVANT CHANGE (problème confirmé) :
# Enhanced Key Usages (EKU) :
#   → Server Authentication (1.3.6.1.5.5.7.3.1)
#   [EKU Remote Desktop Authentication ABSENT = problème confirmé]

# ── RÉSULTAT ATTENDU APRÈS PASSTHROUGH (problème résolu) :
# Enhanced Key Usages (EKU) :
#   → Server Authentication (1.3.6.1.5.5.7.3.1)
#   → Remote Desktop Authentication (1.3.6.1.4.1.311.54.1.2)
#   [EKU présent = connexion NLA 25H2 acceptée]
A2

Vérification via OpenSSL (depuis Linux/WSL ou F5 bash)

Bash — Linux/WSL ou F5 CLI 
# Connexion TLS brute sur le port RDP et extraction du certificat
VIP="10.x.x.x"  # ← Remplacer

echo "" | openssl s_client -connect ${VIP}:3389 -starttls rdp 2>/dev/null | \
    openssl x509 -noout -text | grep -A 20 "X509v3 Extended Key Usage"

# X509v3 Extended Key Usage:
#     TLS Web Server Authentication

# Résultat attendu APRÈS passthrough (cert VDI avec EKU RDP) :
# X509v3 Extended Key Usage:
#     TLS Web Server Authentication
#     1.3.6.1.4.1.311.54.1.2   ← Remote Desktop Authentication présent

# Si openssl -starttls rdp non disponible, dump brut SSL :

B2

Appliquer la configuration Passthrough (FastL4)

Cette séquence de commandes transforme le VS de SSL Bridging (L7) vers SSL Passthrough (L4). Chaque commande est idempotente et peut être rejouée.

F5 TMSH — Change production (ordre strict à respecter) Fenêtre de maintenance requise 
# ── ÉTAPE 1 : Changer le type du VS vers Performance (Layer 4)
tmsh modify ltm virtual <nom-du-vs-rdp> {
    ip-protocol tcp
    profiles replace-all-with { fastL4 { } }
}

# ── ÉTAPE 2 : Supprimer explicitement les profils SSL Client et Server
# (la commande précédente les remplace, mais vérification explicite recommandée)
tmsh modify ltm virtual <nom-du-vs-rdp> \
    profiles delete { rdp-client-ssl rdp-server-ssl tcp }

# ── ÉTAPE 3 : S'assurer que seul fastL4 est attaché
tmsh modify ltm virtual <nom-du-vs-rdp> {
    profiles add { fastL4 { context all } }
}

# ── ÉTAPE 4 : Configurer la persistence Source Address Affinity
tmsh create ltm persistence source-addr rdp-src-affinity {
    timeout 3600
    match-across-services enabled
    match-across-virtuals disabled
}

tmsh modify ltm virtual <nom-du-vs-rdp> {
    persist replace-all-with {
        rdp-src-affinity { default yes }
    }
}

# ── ÉTAPE 5 : Vérifier le health monitor (doit être TCP:3389, pas HTTPS)
tmsh show ltm pool <nom-du-pool-vdi> members detail
# Si monitor est HTTPS ou SSL, le remplacer :
tmsh create ltm monitor tcp rdp-tcp-monitor {
    interval 10
    timeout  31
    dest     *.3389
}
tmsh modify ltm pool <nom-du-pool-vdi> { monitor rdp-tcp-monitor }

# ── ÉTAPE 6 : Sauvegarder la configuration
tmsh save sys config

# ── VÉRIFICATION finale de la configuration résultante
tmsh list ltm virtual <nom-du-vs-rdp>

# Résultat attendu après change :
# ltm virtual <nom-du-vs-rdp> {
#     ip-protocol tcp
#     persist { rdp-src-affinity { default yes } }
#     pool <nom-du-pool-vdi>
#     profiles {
#         fastL4 { }      ← SEUL profil = passthrough actif
#     }
#     [ABSENT : rdp-client-ssl, rdp-server-ssl ← = passthrough confirmé]
# }

Test C — Émettre un certificat avec EKU RDP via PKI AD CS (Option 2)

C1

Créer un template AD CS avec l'OID Remote Desktop Authentication

Sur le serveur AD CS (Autorité de Certification), créer un template de certificat enrichi de l'OID RDP. Comme commander un badge de sécurité à l'imprimerie officielle, en spécifiant qu'il doit inclure l'habilitation "RDP".

PowerShell — Serveur AD CS (Admin CA) Option 2 — PKI 
# ── Créer le template via duplication (méthode recommandée)
# 1. Dans certtmpl.msc : dupliquer "Web Server" → nommer "RDP-Server-F5"
# 2. Onglet Extensions → Application Policies (EKU) → Ajouter manuellement l'OID :

# OID à ajouter au template :
#   Remote Desktop Authentication = 1.3.6.1.4.1.311.54.1.2

# ── Via PowerShell (nécessite module ADCS ou certutil)
# Méthode certutil pour vérifier les templates existants :
certutil -CATemplates

# Vérifier qu'un template avec l'OID RDP existe :
certutil -v -template | Select-String "1.3.6.1.4.1.311.54.1.2" -Context 5

# ── Émettre le certificat depuis le F5 (via CSR)
# Sur le F5, générer un CSR :
tmsh create sys crypto key rdp-f5-key { key-size 2048 key-type rsa-private }
tmsh create sys crypto csr rdp-f5-csr {
    key rdp-f5-key
    common-name "vip-rdp.votre-domaine.com"
    subject-alternative-name "DNS:vip-rdp.votre-domaine.com"
}

# Exporter le CSR :
tmsh list sys crypto csr rdp-f5-csr
# → Copier le CSR et le soumettre à la CA AD CS

# ── Soumettre le CSR à la CA (depuis Windows) :
certreq -submit -attrib "CertificateTemplate:RDP-Server-F5" rdp-f5.csr rdp-f5.cer

# Vérifier que l'EKU RDP est dans le certificat émis :
certutil -dump rdp-f5.cer | Select-String "1.3.6.1.4.1.311.54.1.2"

# Résultat attendu :
# 1.3.6.1.4.1.311.54.1.2 Remote Desktop Authentication
#   [EKU présent dans le certificat émis pour le F5]

# ── Importer le certificat sur le F5 et l'attacher au VS :
tmsh install sys crypto cert rdp-f5-cert from-local-file /path/to/rdp-f5.cer
tmsh modify ltm profile client-ssl rdp-client-ssl { cert-key-chain add { default { cert rdp-f5-cert key rdp-f5-key } } }

D1

Procédure complète de signature via PKI d'entreprise AD CS

PowerShell (Admin) — Poste admin avec accès à la PKI AD CS Solution Problème B 
# ── ÉTAPE 1 : Demander un certificat Code Signing à la PKI AD CS
# (Remplace la création auto-signée si une CA d'entreprise est disponible)
$cert = Get-Certificate `
    -Template "CodeSigning" `
    -CertStoreLocation "cert:\LocalMachine\My"

# OU si auto-signé (lab/test) :
$cert = New-SelfSignedCertificate `
    -Subject "CN=RDP Publisher - Corp VDI" `
    -Type CodeSigningCert `
    -CertStoreLocation cert:\LocalMachine\My `
    -NotAfter (Get-Date).AddYears(3)

# ── ÉTAPE 2 : Récupérer l'empreinte
$thumbprint = $cert.Thumbprint
Write-Host "SHA256 Thumbprint : $thumbprint" -ForegroundColor Cyan

# Vérifier que le certificat a bien l'EKU Code Signing :
$cert.EnhancedKeyUsageList | Format-Table FriendlyName, ObjectId
# Résultat attendu :
# FriendlyName    ObjectId
# Code Signing    1.3.6.1.5.5.7.3.3

# ── ÉTAPE 3 : Signer le ou les fichiers .rdp
$rdpFile = "C:\RDP\vdi-prod.rdp"  # ← Adapter le chemin

rdpsign.exe /v /sha256 $thumbprint $rdpFile

# Résultat attendu :
# Successfully signed: C:\RDP\vdi-prod.rdp
# [Si erreur 0x8009200b : vérifier que la clé privée est accessible]

# ── ÉTAPE 4 : Vérifier la signature
# Inspecter le contenu du .rdp signé (la signature est en bas du fichier)
Get-Content $rdpFile | Select-Object -Last 5
# On doit voir les lignes "signscope:s:..." et "signature:s:..." en fin de fichier

# ── ÉTAPE 5 : Distribuer le certificat public aux clients (GPO)
# Exporter le cert public :
Export-Certificate -Cert $cert -FilePath "C:\RDP\rdp-publisher.cer"

# GPO Computer Config → Windows Settings → Security Settings
# → Public Key Policies → Trusted Publishers → Import rdp-publisher.cer

# OU via GPO "Specify SHA1 thumbprints of trusted .rdp publishers" :

# ── ÉTAPE 6 : Test de validation de la signature

$content = Get-Content $rdpFile -Raw

    Write-Host "✓ Fichier .rdp signé" -ForegroundColor Green

} else {
    Write-Host "✗ Fichier .rdp NON signé" -ForegroundColor Red
}
